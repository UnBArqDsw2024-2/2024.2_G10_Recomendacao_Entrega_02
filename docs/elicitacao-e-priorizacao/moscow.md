# MoSCoW

## Introdução

A técnica MoSCoW é um método amplamente utilizado em gestão de projetos e desenvolvimento de software para priorizar requisitos e funcionalidades. O nome é um acrônimo derivado das palavras Must have (Deve ter), Should have (Deveria ter), Could have (Poderia ter) e Won't have (Não terá por agora), representando diferentes níveis de prioridade atribuídos aos itens analisados.

O principal objetivo do MoSCoW é ajudar equipes e stakeholders a alinhar expectativas e concentrar esforços no que realmente importa para o sucesso de um projeto, especialmente quando os recursos (tempo, orçamento, equipe) são limitados. Cada uma das categorias possui um significado específico:

- Must have (Deve ter)
São itens essenciais e inegociáveis para o sucesso do projeto. Sem eles, o objetivo do projeto não será atingido.

- Should have (Deveria ter)
Representam funcionalidades ou requisitos importantes, mas que não são indispensáveis. Eles podem ser adiados ou substituídos caso o tempo ou os recursos estejam escassos.

- Could have (Poderia ter)
Incluem elementos desejáveis, que agregam valor, mas não são cruciais. Geralmente são implementados se houver tempo ou orçamento disponíveis.

- Won't have (Não terá por agora)
São itens identificados como de baixa prioridade ou que não se alinham com o escopo atual. Apesar disso, eles podem ser revisitados em versões futuras do projeto.

## Metodologia

Para aplicar a técnica MoSCoW e priorizar os requisitos do projeto, nosso grupo organizou uma reunião online utilizando a plataforma Discord. O objetivo principal foi garantir a participação ativa de todos os membros na análise e classificação de cada requisito, promovendo um consenso coletivo.

Um membro do grupo assumiu o papel de moderador, organizando a dinâmica da reunião e garantindo que cada requisito fosse analisado com atenção. O moderador compartilhou sua tela no Discord, exibindo os requisitos para que todos pudessem acompanhar a discussão em tempo real. Cada requisito foi lido em voz alta, e os membros tiveram a oportunidade de expressar suas opiniões sobre sua importância no contexto do projeto.

Durante as discussões, utilizamos a técnica MoSCoW para classificar os requisitos nas categorias:

- Must have (Deve ter)
- Should have (Deveria ter)
- Could have (Poderia ter)
- Won't have (Não terá por agora)

Um segundo membro do grupo ficou responsável por registrar as classificações e colocá-lás no artefato em questão.

## Requisitos Priorizados

Os requisitos utilizados na priorização podem ser vistos no documento de [Requisitos Elicitados](https://unbarqdsw2024-2.github.io/2024.2_G10_Recomendacao_Entrega_02/#/elicitacao-e-priorizacao/requisitos-elicitados?id=requisitos-funcionais).

### Must Have (Deve Ter)

- RF01 - Cadastrar usuário
Necessário para permitir que novos usuários utilizem o sistema, sendo a base para autenticação e funcionalidades dependentes de perfis individuais.

- RF05 - Cadastro de restaurante
Fundamental para que os usuários possam inserir restaurantes no sistema, uma funcionalidade central para o objetivo do projeto.

- RF09 - Login de usuário
Indispensável para autenticar usuários e garantir segurança e personalização dos recursos disponíveis.

- RF10 - Logout de usuário
Essencial para desconectar usuários de maneira segura e evitar acessos indevidos às funcionalidades restritas.

- RF11 - Cadastro de avaliação
Um dos principais propósitos do sistema é permitir que usuários avaliem restaurantes, tornando este requisito imprescindível.

- RF13 - Filtrar por categorias
Facilita a navegação e descoberta de restaurantes, sendo uma funcionalidade indispensável para usabilidade do sistema.

- RF14 - Buscar restaurantes por categorias
Complementa a funcionalidade de filtro, permitindo ao usuário acessar restaurantes de interesse com base em categorias específicas.

- RF17 - Visualizar avaliações por outros usuários
Garante que os usuários possam consultar feedbacks de terceiros, enriquecendo a experiência do sistema e auxiliando na tomada de decisão.

- RF18 - Cadastro único
Evita duplicidade de registros, garantindo a consistência e organização das informações de restaurantes.

- RF19 - Obter média
Proporciona ao usuário uma visão geral da avaliação de um restaurante, um dado essencial para facilitar a escolha.

### Should Have (Deveria Ter)

- RF06 - Editar dados de restaurante
Permite a atualização de informações dos restaurantes, melhorando a qualidade dos dados, mas pode ser implementado após as funcionalidades básicas.

- RF07 - Visualizar dados de restaurante
Oferece detalhes sobre os restaurantes, mas não é imprescindível para o funcionamento inicial do sistema.

- RF12 - Exclusão de avaliação
Embora seja relevante para o controle de conteúdo do usuário, pode ser postergado, pois não compromete a funcionalidade básica.

- RF15 - Buscar restaurantes
Facilita o acesso a restaurantes por nome, mas pode ser adicionado posteriormente sem grande impacto na usabilidade inicial.

- RF16 - Visualizar avaliações por usuário
Uma funcionalidade útil para o controle do próprio usuário, mas não essencial para o funcionamento básico do sistema.

### Could Have (Poderia Ter)


- RF02 - Editar dados da conta de usuário
Permite maior controle ao usuário sobre seus dados, mas não é indispensável no estágio inicial do sistema.

- RF03 - Visualizar dados de conta de usuário
Facilita ao usuário verificar suas informações cadastradas, mas não é crítico para o funcionamento básico.

- RF04 - Excluir conta de usuário
Garante maior controle ao usuário, mas pode ser postergado, uma vez que não interfere no funcionamento principal.

### Won't Have (Não Terá por Agora)

- RF08 - Excluir restaurante cadastrado
Sua ausência inicial não compromete o funcionamento do sistema, já que outras funcionalidades mais importantes devem ser priorizadas.

- RF20 - Curtir avaliação
Um recurso interessante, mas secundário e não essencial para o funcionamento básico do sistema.

- RF21 - Ordenar avaliação
Melhora a experiência do usuário, mas não é necessário para o lançamento inicial do sistema.
